Architectural Weakness Analysis
Weakness 1 — Docker CLI is woven into ongoing monitoring (Critical)

All three Docker-managed components (Docker daemon itself, Redis, RabbitMQ) resolve their status via CLI sub-process execution on every 8-second background refresh:

docker info (Docker status)
docker inspect --format "{{.State.Status}}" sme-redis (Redis status)
docker inspect --format "{{.State.Status}}" sme-rabbitmq (RabbitMQ status)
When Docker Desktop is not running (the common cold-start state), each of these commands hangs until its 1500ms timeout fires. The three calls are parallelized inside gatherStatus(), but the blocking characteristic remains — and the result is misleading 'warning' statuses rather than a clean 'stopped'.

The native alternative for Redis is a raw TCP + Redis protocol PING (net.Socket → send *1\r\n$4\r\nPING\r\n, expect +PONG). For RabbitMQ the management HTTP API at http://localhost:15672/api/health/checks/aliveness/%2F returns {"status":"ok"} with no CLI involved. Docker itself has no universally-available pure-Node alternative, but its check could be gated so it only runs when a container-based check is already inconclusive.

Weakness 2 — netstat -ano used for all 8 backend/frontend service status decisions

All eight NestJS/Next.js services set their status field based on whether pidMap[port] is populated — i.e., whether netstat -ano found a LISTENING TCP entry. This is doubly inefficient:

It runs a full OS-level netstat -ano (enumerating all connections) on every refresh cycle, just to find 8 specific ports.
The status derived from it is redundant: if the HTTP /health probe (Layer 7) already returns 200, the service is obviously running. If it returns null/timeout, it is stopped. netstat adds no information that the HTTP probe doesn't already provide, and it is far more expensive.
The native Node.js alternative is net.createServer().listen(port): if it throws EADDRINUSE, the port is occupied. Or — cleaner — simply derive status directly from the HTTP health probe result, eliminating the port-scan pass entirely for these services.

Weakness 3 — Frontend health probe endpoints return 404 (High)

probeHttp(3101, '/health') and probeHttp(3102, '/health') are called on web-admin and web-portal while running. Next.js has no built-in /health route. The result is health: 'HTTP 404' displayed permanently on the dashboard while the app is healthy. The status shows running (from netstat) but health shows HTTP 404 — a contradiction that erodes trust in the dashboard.

The correct probe for a Next.js app is either GET / (which returns 200 or 307) or a purpose-built pages/api/health.ts route returning { ok: true }.

Weakness 4 — PostgreSQL ongoing monitoring is Layer 4 only (Medium)

Boot verification correctly uses psql -c 'SELECT 1' to confirm the engine is fully accepting queries. The ongoing dashboard monitor only does net.Socket.connect(:5432) (TCP). PostgreSQL's port can be open while the engine is still replaying WAL after a crash ("database system is starting up"), or in a long checkpoint. A TCP OK on the dashboard would falsely show running and health: TCP OK in such a state.

Since sme-monitor.js has no npm dependencies, a SELECT 1 check requires spawning psql — acceptable as a lower-frequency supplementary check (e.g., every 30s) separate from the 8s refresh loop.

Weakness 5 — Prisma client monitoring checks directory, not the DLL binary (Low)

fs.existsSync('apps/iam-service/src/generated/prisma-client') confirms the directory exists. The actual artifact that fails (and that sme-fullboot.ps1 correctly validates post-generate) is query_engine-windows.dll.node. A directory can persist from a previous generate while the DLL itself is missing or zero-bytes after a failed generation. The monitor should check fs.existsSync('.../query_engine-windows.dll.node') for a truthful generated vs missing status.

Weakness 6 — detectInfraVersion() runs CLI commands on every background refresh cycle (Low)

docker version, docker exec sme-redis redis-cli info server, docker exec sme-rabbitmq rabbitmqctl version, and psql --version all fire every 8 seconds in the background cache loop. These are not time-critical: infrastructure versions do not change between refreshes. This code already has a 6-hour TTL cache (LATEST_VER_CACHE) for the remote latest-version fetches — but there is no TTL for the local installed versions. A separate long-TTL cache (e.g., 10 minutes, or fetch-once at startup) for local version detection would eliminate these CLI forks entirely after the first cycle.

The boot path (sme-fullboot.ps1) is architecturally sound — every component gets a proper Layer 7 or native readiness proof before the next stage starts. The monitoring path (sme-monitor.js) is where the gap lies: the status signal for 10 of 12 components still relies on either Docker CLI inspection or a netstat text-parse rather than direct native probes.
